# /serena:reset - Serenaメモリ管理ガイド

## 📋 現在のメモリ確認

### ステップ1: メモリ一覧を表示
```
"Serenaメモリの一覧を見せて"
```

### ステップ2: 内容を確認
```
"project_overview メモリの内容を見せて"
```

---

## 🗑️ メモリ削除方法

### 方法1: 個別削除（選択的クリーンアップ）

**タスク状態メモリを削除**:
```
"task_completion_checklist メモリを削除して"
"調査結果の古いメモリを削除して"
```

**推奨する削除対象**:
- タスクチェックリスト
- 一時的な調査結果（日付付き）
- 完了したタスクの進捗情報
- セッション固有の状態

---

### 方法2: カテゴリ別削除

**タスク関連のみ削除**:
```
"タスク状態に関するSerenaメモリを全て削除して"
```

**古い調査結果を削除**:
```
"2ヶ月以上前の調査メモリを削除して"
```

---

### 方法3: 全削除して再スタート

**完全リセット**:
```
"sample-reference-projectプロジェクトのSerenaメモリを全て削除して"
```

その後、再オンボーディング:
```
"このプロジェクトを再オンボーディングして"
```

---

## 🔄 再オンボーディング手順

### 完全な再オンボーディング

**ステップ1: 既存メモリを確認**
```
"Serenaメモリを一覧表示"
```

**ステップ2: 不要なメモリを削除**
```
"タスク状態と一時調査結果のメモリを削除"
```

**ステップ3: プロジェクト再分析**
```
"sample-reference-projectプロジェクトのアーキテクチャと構造を分析してSerenaメモリに保存して"
```

---

## ✅ 保存すべきメモリ（コードベース理解）

Serenaには**コード関連情報のみ**保存:

### 推奨するメモリ内容:
- ✅ プロジェクト構造 (project_structure)
- ✅ アーキテクチャパターン (architecture_patterns)
- ✅ コードスタイル規約 (code_conventions)
- ✅ 技術スタック情報 (tech_stack)
- ✅ 主要コンポーネント関係 (component_relationships)
- ✅ オンボーディング情報 (onboarding_guide)

---

## ❌ 保存すべきでないメモリ（タスク/セッション状態）

以下はSerenaに保存しない:

### 削除推奨:
- ❌ タスクチェックリスト → TodoWrite
- ❌ Issue管理情報 → GitHub
- ❌ セッション進捗 → Session Context
- ❌ 一時的な調査結果 → 必要ならファイルに保存
- ❌ ビルド/テスト結果 → ログファイル
- ❌ 日付付きの調査メモ → claudedocs/

---

## 🔍 メモリ内容の分類例

### 現在のメモリを確認して分類:

```
"Serenaメモリを一覧表示して、各メモリが
 コードベース理解かタスク状態か分類して"
```

### 分類結果に基づいて削除:

```
"タスク状態のメモリだけ削除して、
 コードベース理解のメモリは残して"
```

---

## 💡 ベストプラクティス

### 定期的なクリーンアップ（推奨: 月1回）

1. **メモリ一覧確認**
2. **タスク状態メモリを削除**
3. **古い調査結果（2ヶ月以上）を削除**
4. **コードベース理解メモリのみ保持**

### プロジェクト開始時

```
"新しいプロジェクトをSerenaにオンボーディング:
 - プロジェクト構造を分析
 - アーキテクチャパターンを抽出
 - コードスタイルを理解
 - 主要コンポーネントを把握"
```

### プロジェクト終了時

```
"完了したプロジェクトのSerenaメモリを削除"
```

---

## 🚨 注意事項

### メモリ削除前の確認

必ず内容を確認してから削除:
```
"削除前に [メモリ名] の内容を確認したい"
```

### 復元不可

削除したメモリは復元できません。重要な情報は:
- claudedocs/ にMarkdownで保存
- Git commitメッセージに記録
- ドキュメントファイルとして永続化

---

## 📚 関連コマンド
