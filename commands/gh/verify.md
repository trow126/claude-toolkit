# GitHub Issue連携ログ分析コマンド

ログファイルを分析し、修正検証・異常検出・デバッグ支援を行い、結果をGitHub Issueコメントとして投稿します。

## 基本構文

```bash
/gh:verify <log_path> [options]

Options:
  --issue NUMBER          # 明示的にIssue番号を指定
  --mode <verify|detect|debug|auto>  # 分析モード（デフォルト: auto）
  --context "description" # 手動でコンテキストを指定
  --dry-run              # GitHubに投稿せず結果を表示のみ
  --verbose              # 詳細な分析結果を含める
```

---

## 分析モード

| Mode | 日本語 | 用途 |
|------|--------|------|
| `verify` | 修正検証 | 直前の修正がログに正しく反映されているか確認 |
| `detect` | 異常検出 | 正常終了しているが違和感・パターン異常がないか分析 |
| `debug` | デバッグ支援 | エラーログから問題箇所を特定、Issue化を支援 |
| `auto` | 自動判定 | ログ内容からモードを自動選択（デフォルト） |

---

## 使用例

### 修正検証（verify）

Issue作業後、修正が正しく反映されているか確認：

```bash
# 現在のIssueコンテキストを使用
/gh:verify /var/log/app.log --mode verify

# 明示的にIssue指定
/gh:verify /var/log/app.log --issue 42 --mode verify
```

### 異常検出（detect）

正常終了しているが違和感がないかチェック：

```bash
/gh:verify /var/log/app.log --mode detect
```

**検出する異常**:
- メッセージの異常頻度（リトライ過多など）
- 予期しない操作順序
- 期待されるエントリの欠如
- タイミング異常（処理時間の遅延など）
- 微妙なエラー指標（deprecation警告など）

### デバッグ支援（debug）

エラーログから問題箇所を特定：

```bash
/gh:verify /var/log/error.log --issue 42 --mode debug --verbose
```

### ドライラン

GitHubに投稿せず結果を確認：

```bash
/gh:verify /tmp/test.log --dry-run
```

## 処理フロー (6 Phases)

### Phase 0: Context Extraction
**目的**: 修正コンテキストを収集

**優先順位**:
1. `--issue NUMBER` フラグ（明示指定）
2. Serena Memory の active checkpoint (`issue_*_checkpoint`)
3. 会話履歴からの推論（ファイル編集、エラーメッセージ、関数名）
4. `--context "..."` フラグ（手動）

**検証**: Issue が見つからず `--dry-run` でない場合、Issue 番号を要求

### Phase 1: Log Ingestion
**目的**: ログファイルの読み込みと特性把握

1. ファイル存在・読み取り可能性を検証
2. ファイルサイズ確認（>10MB の場合、末尾10000行に制限）
3. 特性検出:
   - タイムスタンプ形式
   - ログレベル (INFO/WARN/ERROR)
   - 構造タイプ（非構造化/半構造化/JSON）
4. 論理エントリへの前処理（マルチラインスタックトレース対応）

### Phase 2: Analysis (Mode-Specific)

**verify モード**:
- コンテキストからキーワード抽出
- 変更されたコンポーネントの呼び出しを検索
- 期待される動作パターンの出現確認
- 関連エラーの不在確認
- タイムライン検証（タイムスタンプがある場合）

**detect モード**:
- パターンベースラインの確立（繰り返しパターン、頻度）
- 異常検出:
  - メッセージの異常頻度
  - 予期しない操作順序
  - 期待されるエントリの欠如
  - タイミング異常
  - 微妙なエラー指標（リトライ、deprecation）

**debug モード**:
- 完全なスタックトレース付きエラー抽出
- タイプ別エラーグループ化と発生回数カウント
- トリガーイベントへのエラーシーケンス追跡
- 先行する警告の検出
- Issue 対応コンテンツの準備

**auto モード判定ロジック**:
```
if error_rate > 5%:
    mode = "debug"
elif has_context_keywords and recent_changes:
    mode = "verify"
else:
    mode = "detect"
```

### Phase 3: Correlation
**目的**: 分析結果とコード変更/Issue コンテキストの関連付け

- 分析結果とコンテキストキーワードのマッチング
- 最近のファイル変更との相互参照
- タイムライン整合性（変更タイムスタンプ vs ログエントリ）
- Serena シンボル検索で参照コードを確認

### Phase 4: Synthesis
**目的**: 構造化された分析レポート生成

コンポーネント:
- サマリー（2-3文）
- 統計（エントリ数、エラー率）
- 主要な発見（モード固有）
- 推奨事項（アクション可能な項目）
- 関連コンテキスト（Issue/変更へのリンク）

### Phase 5: Publication
**目的**: GitHub Issue へ分析結果を投稿

1. `--dry-run` の場合: コンソール表示のみ
2. `gh issue comment NUMBER --body "..."` でコメント投稿
3. ローカルバックアップを `claudedocs/analysis/` に保存（オプション）

use log-verifier skill to analyze <log_path>

## 大ファイル対応

10MB超のログファイルは自動的に末尾10000行に制限されます：

```bash
# 内部処理
tail -n 10000 /path/to/large.log > /tmp/analysis_target.log
```

**理由**: 直近のログが修正検証に最も関連するため

---

## エラー処理

| シナリオ | 対応 |
|----------|------|
| ファイルが見つからない | パスを含むエラーメッセージ |
| ファイルが大きすぎる | 末尾10000行に制限、警告表示 |
| Issue関連付けなし | `--dry-run`でない場合、Issue番号を要求 |
| 分析結果なし | "クリーン" ステータスを報告 |
| バイナリファイル | 検出して中止 |
| GitHub APIエラー | 1回リトライ後、ローカル保存にフォールバック |

## ワークフロー統合

### Issue作業フローとの連携

```bash
# 1. Issue作業開始
/gh:issue work 42

# 2. 修正実装
# ... コード編集 ...

# 3. アプリケーション実行
# ... テスト実行 ...

# 4. ログ検証
/gh:verify /var/log/app.log --mode verify
# → 自動的にIssue #42に分析結果をコメント投稿

# 5. Issue完了
/gh:issue close 42
```

### 本番障害調査フロー

```bash
# 1. エラーログ分析
/gh:verify /var/log/error.log --mode debug --dry-run

# 2. 問題を確認したらIssue作成
/gh:issue create "Bug: Connection timeout in AuthService"

# 3. 詳細分析をIssueに紐付け
/gh:verify /var/log/error.log --issue 99 --mode debug --verbose
```

## 実行指示

**あなたは今、`/gh:verify` コマンドを実行しています。**

以下の手順で実行してください：

### Step 1: コンテキスト抽出 (Phase 0)

1. `--issue NUMBER` が指定されている場合、そのIssueを使用
2. なければ Serena Memory からチェックポイント確認:
   ```
   list_memories() → "issue_*_checkpoint" を検索
   ```
3. なければ会話履歴から推論（編集したファイル、関連キーワード）
4. `--dry-run` でなくIssueが見つからない場合、ユーザーにIssue番号を確認

### Step 2: ログ読み込み (Phase 1)

1. ファイル存在確認:
   ```bash
   ls -la <log_path>
   ```
2. サイズ確認（10MB超の場合は末尾10000行に制限）:
   ```bash
   wc -l <log_path>
   tail -n 10000 <log_path>  # 大ファイルの場合
   ```
3. バイナリチェック:
   ```bash
   file <log_path>
   ```

### Step 3: モード判定と分析 (Phase 2)

**auto モードの場合**:
```bash
# エラー率計算
error_count=$(grep -c -E "ERROR|Exception|Fatal|CRITICAL" <log_path>)
total_count=$(wc -l < <log_path>)
# error_rate > 5% → debug
# context_keywords found → verify
# else → detect
```

**モード別分析**:

- **verify**: Grep でコンテキストキーワード検索、期待パターン確認
- **detect**: パターン頻度分析、異常検出
- **debug**: エラー抽出、スタックトレース収集

### Step 4: レポート生成 (Phase 4)

上記「出力フォーマット」に従ってMarkdownレポートを生成

### Step 5: 投稿 (Phase 5)

`--dry-run` の場合:
- レポートをコンソール表示のみ

それ以外:
```bash
gh issue comment <NUMBER> --body "$(cat <<'EOF'
[生成したレポート]
EOF
)"
```
