# GitHub Issue連携ログ分析コマンド

ログファイルを分析し、修正検証・異常検出・デバッグ支援を行い、結果をGitHub Issueコメントとして投稿します。

**🚀 v2.0: サブエージェント委譲によるコンテキスト効率化**
- ログ分析処理をgeneral-purposeサブエージェントに委譲
- 大量ログ（10000行）をメインコンテキストに読み込まない
- 90%のコンテキスト削減を実現（50K+ → ~3K tokens）

## 基本構文

```bash
/gh:verify <log_path> [options]

Options:
  --issue NUMBER          # 明示的にIssue番号を指定
  --mode <verify|detect|debug|auto>  # 分析モード（デフォルト: auto）
  --context "description" # 手動でコンテキストを指定
  --dry-run              # GitHubに投稿せず結果を表示のみ
  --verbose              # 詳細な分析結果を含める
```

---

## 分析モード

| Mode | 日本語 | 用途 |
|------|--------|------|
| `verify` | 修正検証 | 直前の修正がログに正しく反映されているか確認 |
| `detect` | 異常検出 | 正常終了しているが違和感・パターン異常がないか分析 |
| `debug` | デバッグ支援 | エラーログから問題箇所を特定、Issue化を支援 |
| `auto` | 自動判定 | ログ内容からモードを自動選択（デフォルト） |

## 使用例

### 修正検証（verify）

Issue作業後、修正が正しく反映されているか確認：

```bash
# 現在のIssueコンテキストを使用
/gh:verify /var/log/app.log --mode verify

# 明示的にIssue指定
/gh:verify /var/log/app.log --issue 42 --mode verify
```

### 異常検出（detect）

正常終了しているが違和感がないかチェック：

```bash
/gh:verify /var/log/app.log --mode detect
```

**検出する異常**:
- メッセージの異常頻度（リトライ過多など）
- 予期しない操作順序
- 期待されるエントリの欠如
- タイミング異常（処理時間の遅延など）
- 微妙なエラー指標（deprecation警告など）

### デバッグ支援（debug）

エラーログから問題箇所を特定：

```bash
/gh:verify /var/log/error.log --issue 42 --mode debug --verbose
```

### ドライラン

GitHubに投稿せず結果を確認：

```bash
/gh:verify /tmp/test.log --dry-run
```

## 処理フロー (6 Phases) - v2.0

### Phase 0: Context Extraction（メインコンテキスト）
**目的**: 修正コンテキストを収集

**優先順位**:
1. `--issue NUMBER` フラグ（明示指定）
2. Serena Memory の active checkpoint (`issue_*_checkpoint`)
3. 会話履歴からの推論（ファイル編集、エラーメッセージ、関数名）
4. `--context "..."` フラグ（手動）

**検証**: Issue が見つからず `--dry-run` でない場合、Issue 番号を要求

### Phase 1-4: Subagent Delegation（🚀 サブエージェント委譲）
**目的**: ログ分析処理を独立コンテキストで実行

```yaml
Task(
  subagent_type: "general-purpose",
  model: "sonnet",
  prompt: |
    ## ログ分析タスク

    **分析対象**: {log_path}
    **モード**: {mode}
    **コンテキスト**: {context_keywords}

    **Phase 1: Log Ingestion**
    1. ファイル読み込み: Read({log_path})
    2. サイズ確認: 10000行超の場合は末尾10000行
    3. 特性検出: タイムスタンプ、ログレベル、構造タイプ

    **Phase 2: Analysis ({mode})**
    {mode_specific_instructions}

    **Phase 3: Correlation**
    - コンテキストキーワードとのマッチング
    - 異常パターンの関連付け

    **Phase 4: Synthesis**
    以下の形式でMarkdownレポートを生成:

    ## 📊 Log Analysis Report

    **File**: {log_path}
    **Mode**: {mode}
    **Analyzed**: {timestamp}

    ### 🎯 Summary
    [2-3文]

    | Metric | Value |
    |--------|-------|
    | Total Entries | N |
    | Errors | N (X%) |
    | Warnings | N (X%) |

    ### 🔍 Key Findings
    [分析結果]

    ### 📋 Recommendations
    - [ ] アクション項目

    _🤖 Generated by `/gh:verify`_

    **⚠️ 返却形式**: 上記Markdownレポートのみ返却
)
```

**サブエージェント内で実行される処理**:
- ログファイル全体の読み込み（Read tool）
- パターン検索（Grep tool）
- 統計計算
- 異常検出アルゴリズム
- レポート生成

### Phase 5: Publication（メインコンテキスト）
**目的**: GitHub Issue へ分析結果を投稿

1. サブエージェントからレポートを受信（~2000 tokens）
2. `--dry-run` の場合: コンソール表示のみ
3. それ以外: `gh issue comment NUMBER --body "..."` で投稿
4. ローカルバックアップを `claudedocs/analysis/` に保存（オプション）

## 大ファイル対応

10MB超のログファイルは自動的に末尾10000行に制限されます：

```bash
# 内部処理
tail -n 10000 /path/to/large.log > /tmp/analysis_target.log
```

**理由**: 直近のログが修正検証に最も関連するため

---

## エラー処理

| シナリオ | 対応 |
|----------|------|
| ファイルが見つからない | パスを含むエラーメッセージ |
| ファイルが大きすぎる | 末尾10000行に制限、警告表示 |
| Issue関連付けなし | `--dry-run`でない場合、Issue番号を要求 |
| 分析結果なし | "クリーン" ステータスを報告 |
| バイナリファイル | 検出して中止 |
| GitHub APIエラー | 1回リトライ後、ローカル保存にフォールバック |

## 実行指示 - v2.0 サブエージェント委譲パターン

**あなたは今、`/gh:verify` コマンドを実行しています。**

### 🚀 v2.0 実行フロー（コンテキスト効率化）

**v2.0ではログ分析処理をgeneral-purposeサブエージェントに委譲し、メインコンテキストを軽量に保ちます。**

### Step 1: コンテキスト抽出（メインコンテキスト・軽量）

```yaml
優先順位で Issue 特定:
1. --issue NUMBER が指定 → そのIssueを使用
2. Serena Memory確認:
   list_memories() → "issue_*_checkpoint" 検索
3. 会話履歴から推論（編集ファイル、キーワード）
4. --dry-run でなくIssue未特定 → ユーザーに確認

結果:
  context = {
    "issue_number": N or null,
    "keywords": ["keyword1", "keyword2"],
    "recent_changes": ["file1.py", "file2.py"]
  }
```

### Step 2: モード判定（メインコンテキスト・軽量）

```yaml
auto モードの場合、簡易判定:
  - Bash: file <log_path>  # バイナリチェック
  - Bash: wc -l <log_path>  # 行数確認

モード自動判定はサブエージェントに委譲:
  - サブエージェント内でエラー率計算
  - サブエージェント内でモード決定
```

### Step 3: サブエージェント起動（🔴 必須）

```yaml
Task(
  subagent_type: "general-purpose",
  model: "sonnet",
  prompt: |
    ## ログ分析タスク

    **分析対象**: {log_path}
    **モード**: {mode} (auto の場合は自動判定)
    **Issue**: #{issue_number}
    **コンテキストキーワード**: {keywords}
    **最近の変更**: {recent_changes}

    **実行手順**:

    1. **ログ読み込み**
       - Read({log_path}) でファイル読み込み
       - 10000行超の場合は末尾10000行に制限
       - バイナリファイルの場合はエラー

    2. **モード自動判定** (mode=auto の場合)
       - エラー率 > 5% → debug
       - コンテキストキーワードあり → verify
       - それ以外 → detect

    3. **分析実行**
       - verify: キーワード検索、期待パターン確認
       - detect: 頻度分析、異常検出
       - debug: エラー抽出、スタックトレース収集

    4. **レポート生成**
       以下のMarkdown形式で返却:

    ## 📊 Log Analysis Report

    **File**: `{log_path}`
    **Mode**: {detected_mode}
    **Analyzed**: {timestamp} JST
    **Log Period**: {start_time} - {end_time}

    ---

    ### 🎯 Summary

    [2-3文の簡潔なサマリー]

    | Metric | Value |
    |--------|-------|
    | Total Entries | N |
    | Errors | N (X%) |
    | Warnings | N (X%) |
    | Related to Issue | N entries |

    ---

    ### 🔍 Key Findings

    [モード固有の詳細分析結果]

    ---

    ### 📋 Recommendations

    - [ ] Action item 1
    - [ ] Action item 2

    ---

    ### 📎 Related Context

    - Issue #{issue_number}: "{issue_title}"
    - Recent changes: {files}

    ---

    _🤖 Generated by `/gh:verify`_

    **⚠️ 返却形式**: 上記Markdownレポートのみ返却。ログ生データは含めない。
)
```

### Step 4: 結果受信と投稿（メインコンテキスト）

```yaml
サブエージェントからレポートを受信:
  - Markdownレポートのみ（~2000 tokens）
  - ログ生データは含まれない

投稿処理:
  --dry-run の場合:
    - レポートをコンソール表示のみ

  それ以外:
    gh issue comment {issue_number} --body "$(cat <<'EOF'
    {received_report}
    EOF
    )"
```
